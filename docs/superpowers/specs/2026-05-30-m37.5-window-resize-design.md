# M37.5 — Window resize correctness (Design)

**Date:** 2026-05-30
**Milestone:** M37.5 (bugfix follow-up to M37; sits between M37 and M38)
**Prerequisite:** M30+ editor module; M37 World (in main but irrelevant to this fix).

## Goal

Fix a pre-existing editor bug surfaced during M37 visual verification: resizing
the sandbox window shifts mouse picking and gizmo hit-tests, and the renderer
keeps rendering to the original 1280×720 surface scaled to whatever size the OS
window has become.

After M37.5, the sandbox renders at the window's actual framebuffer size, the
projection-matrix aspect ratio updates on resize, mouse picking computes correct
screen-to-world rays for the live dimensions, and minimizing the window does
not crash (the Vulkan branch already handles 0×0 swapchain — we just need to
stop calling into it on minimized frames).

## Root cause

Three independent code paths each cache the original window dimensions and
never update them:

1. **`engine/core/Window.cpp:17` — `framebufferSizeCallback`**: the Vulkan
   branch is a no-op (`(void)w; (void)h;` with a comment deferring to
   `Renderer::setViewport`, which is never called in response).
2. **`games/11-sandbox/main.cpp`**: hard-coded compile-time constants
   `kScreenW = 1280` / `kScreenH = 720` are used at multiple sites:
   - `app.cfg.width/height` at construction (line 243-244) — fine, that's the
     initial size,
   - `renderer.setViewport(kScreenW, kScreenH)` (line 257) — set once,
   - projection-matrix `aspect = kScreenW / kScreenH` (line 437) — frozen,
   - `screenPointToRay(..., Vec2{kScreenW, kScreenH}, ...)` for picking
     (line 575-580) — frozen.
3. **`Renderer::setViewport`** already implements a complete pending-resize
   pipeline (`VulkanRenderer.cpp:1085`): queues `pendingResize_`, consumed in
   `beginFrame` with `vkDeviceWaitIdle` + `swapchain_.recreate` +
   `pipelines_.recreateFramebuffers` + `postProcess_.resize`, with a 0×0
   guard and `VK_ERROR_OUT_OF_DATE_KHR` fallback. Nobody calls it on resize.

The fix is therefore a plumbing problem, not a Vulkan problem.

## Scope

**In scope:**
- `Window::framebufferSizeCallback` Vulkan branch updates Window's
  `width_`/`height_` members and sets a `resized_` flag.
- `Window::consumeResized()` returns true once per resize event, clears the
  flag.
- Sandbox host queries `app.window().width()/height()` for the live framebuffer
  dimensions instead of `kScreenW`/`kScreenH`.
- Sandbox host's main loop, before each `renderer.beginFrame`:
  1. If `window.consumeResized()` returned true, call
     `renderer.setViewport(window.width(), window.height())` and rebuild the
     projection matrix from the new aspect.
  2. If `window.width() == 0` OR `window.height() == 0` (minimized), call
     `glfwWaitEvents()` (block until the OS sends a real event) and `continue`
     to the next loop iteration — skip the whole frame.
- Picking call site (line 575-580) uses live `app.window().width()/height()`.
- The compile-time `kScreenW`/`kScreenH` constants are renamed to
  `kInitialW`/`kInitialH` (their actual purpose) — used only for the initial
  Window construction.

**Out of scope:**
- HiDPI logical-vs-physical pixel separation. Current desktop Windows install is
  not HiDPI; GLFW's `glfwGetFramebufferSize` and `glfwGetWindowSize` return the
  same value here. Defer until needed.
- DPI-aware mouse-coordinate scaling.
- Multi-window, fullscreen toggle, persistent window-size restore.
- Event-driven `VK_ERROR_OUT_OF_DATE_KHR`-only path. The renderer already has
  this as a safety net; the polled `consumeResized()` is the primary signal
  because GLFW's framebuffer-size callback is more reliable than waiting for
  Vulkan to detect the surface change.
- Moving projection-matrix rebuild into the Renderer. Projection lives with the
  camera/host; correct seam.
- Window observer pattern / event hooks. Single consumer (the sandbox) — YAGNI.
- `Application` changes to centralize resize handling. The sandbox owns the
  main loop; adding plumbing through Application is premature abstraction.
- Touching `Picking`, `Gizmo`, `Inspector`, `Outliner`. Picking's signature is
  unchanged; the gizmo has no direct screen-dimension references (verified by
  grep); the others are unaffected.

## Architecture

### `engine/core/Window`

```cpp
// engine/core/Window.h — only the new bits shown
class Window {
public:
    int  width()  const { return width_; }
    int  height() const { return height_; }
    bool consumeResized();          // NEW
private:
    // existing: handle_, width_, height_, ...
    bool resized_ = false;          // NEW
};
```

```cpp
// engine/core/Window.cpp — only the changed bits shown
namespace {
void framebufferSizeCallback(GLFWwindow* w, int width, int height) {
    auto* self = static_cast<Window*>(glfwGetWindowUserPointer(w));
    if (!self) return;
    self->width_  = width;
    self->height_ = height;
    self->resized_ = true;
}
}  // namespace

bool Window::consumeResized() {
    if (!resized_) return false;
    resized_ = false;
    return true;
}
```

Confirm that `Window::Window` calls `glfwSetWindowUserPointer(handle_, this)`
before installing the framebuffer callback. If not, add that line — without
it, the callback can't reach back into the Window instance to update its
fields.

If `width_`/`height_` are currently `private`, expose them to the
file-local `framebufferSizeCallback` via either:
- `friend void framebufferSizeCallback(...);` in the class declaration, OR
- a small `void onFramebufferResize(int w, int h)` member that the callback
  calls through a pointer.

Implementer should pick the smaller diff for the existing style.

### `games/11-sandbox/main.cpp`

Five concrete edits:

1. Rename `kScreenW`/`kScreenH` to `kInitialW`/`kInitialH` so the constants'
   actual purpose ("initial window size at construction") is honest. The
   `cfg.width`/`cfg.height` assignments at lines 243-244 use these.

2. Replace `renderer.setViewport(kScreenW, kScreenH)` (line 257) with
   `renderer.setViewport(app.window().width(), app.window().height())`.

3. Replace the const `aspect`/`proj` computation (line 437) with a lambda
   that's also callable on resize:
   ```cpp
   auto computeProj = [&]() {
       const float aspect = static_cast<float>(app.window().width())
                          / static_cast<float>(app.window().height());
       return iron::perspective(kFovDeg, aspect, kNear, kFar);
   };
   iron::Mat4 proj = computeProj();
   ```
   (Use the existing `kFovDeg`/`kNear`/`kFar` constant names — they're already
   in the file.)

4. At the top of the main loop body (inside the `app.setUpdate(...)`
   callback), before any GPU work:
   ```cpp
   if (app.window().consumeResized()) {
       const int w = app.window().width();
       const int h = app.window().height();
       renderer.setViewport(w, h);
       if (w > 0 && h > 0) proj = computeProj();
   }
   if (app.window().width() == 0 || app.window().height() == 0) {
       return;   // setUpdate callback — skip this frame.
   }
   ```
   The exact early-return shape depends on whether `setUpdate` is a callback
   (return; ends the frame) or an explicit loop (continue;). Implementer
   confirms by reading the file.

5. Replace the picking screen-size argument (line 575-580):
   ```cpp
   iron::screenPointToRay(view, proj, mousePx,
       iron::Vec2{static_cast<float>(app.window().width()),
                  static_cast<float>(app.window().height())},
       ...);
   ```

### Picking / Gizmo / Inspector / Outliner — UNTOUCHED

- `screenPointToRay(view, proj, mousePx, screenSize, ...)`: signature
  unchanged. Only the caller's `screenSize` argument changes.
- Gizmo has no direct screen-dimension references (verified). Inherits the
  picking fix for free.
- Inspector / Outliner don't reference screen dimensions.

### Renderer (Vulkan backend) — UNTOUCHED

`VulkanRenderer::setViewport`, the pending-resize consumption in `beginFrame`,
the 0×0 swapchain guard, and the `VK_ERROR_OUT_OF_DATE_KHR` fallback all
already work. M37.5 just makes them get called.

## Data flow

```
GLFW resize event
  → framebufferSizeCallback                       (engine/core/Window.cpp)
      → Window::width_/height_/resized_ updated
  → glfwPollEvents returns                        (engine/core/Application.cpp)
  → host setUpdate callback runs                  (games/11-sandbox/main.cpp)
      → if (window.consumeResized()):
            renderer.setViewport(w, h)             — queues pendingResize_
            proj = computeProj()                   — new aspect
      → if (window size == 0):
            return                                 — skip frame
      → ... game logic ...
  → renderer.beginFrame()                          (engine/render/backends/vulkan/VulkanRenderer.cpp)
      → if (pendingResize_):
            vkDeviceWaitIdle
            swapchain_.recreate(w, h)
            pipelines_.recreateFramebuffers
            postProcess_.resize
            pendingResize_ = false
      → vkAcquireNextImageKHR
      → ... normal frame ...
```

## Testing

No new unit tests. The Window resize signal depends on GLFW callbacks fired by
the windowing system; mocking that costs more than the test is worth. The
existing `test_picking` covers `screenPointToRay` math; the signature doesn't
change. Visual gate is the test.

**Visual gate checklist** (the implementer hands off to the user):

| Action | Expected |
|---|---|
| Resize window larger (drag corner) | Render fills new size; click-pick hits the right entity; gizmo handles track the cursor |
| Resize window smaller | Same |
| Minimize and restore | No crash on minimize; renderer resumes cleanly on restore |
| Click + drag a gizmo handle after a resize | Drag follows cursor exactly (no offset) |
| Switch Selection Effect after resize | M36 effect renders correctly at new framebuffer size |

## Success criteria

1. Resizing the editor window updates the rendered framebuffer to match.
2. Mouse picking hits the right entity at any window size.
3. Gizmo handle picking and tangent-drag tracking follow the cursor at any
   window size.
4. Minimizing the editor does not crash; restoring it resumes cleanly.
5. 48 / 48 tests still green (no test regressions — this is a sandbox + window
   plumbing fix).
6. Zero changes to: Renderer backend, Picking, Gizmo, editor panels.

## Risks

| Risk | Mitigation |
|---|---|
| Window's `framebufferSizeCallback` doesn't get called for the OpenGL backend after the rename — could break OpenGL builds | Keep the existing OpenGL `glViewport` branch; the new Window-state update applies to BOTH branches (size + flag are useful for either backend). OpenGL builds untested by CI but should compile. |
| Window's `width_`/`height_` accessed from main-loop concurrently with GLFW callback firing on event poll | Single-threaded — GLFW callbacks run inside `glfwPollEvents` on the main thread. No race. |
| `glfwSetWindowUserPointer` is not actually called today, so the callback can't reach Window state | If true, add it in `Window::Window` immediately after window creation. Trivial. |
| Some other call site in the sandbox uses `kScreenW`/`kScreenH` and we miss it | grep before committing: `grep -n "kScreen\(W\|H\)" games/11-sandbox/main.cpp` should return zero hits after the rename + replace. |

After M37.5: foundation track resumes with M38 (reflection / type registry).
