# Roadmap: Prefabs ‚Üí Spawning ‚Üí Navigation ‚Üí AI (M70‚ÄìM73)

**Date:** 2026-06-14
**Status:** Agreed (direction set with Elias)

## Context

M69 (scene hierarchy) just merged: `SceneEntity` has `parentIndex`, and
`SceneHierarchy` provides tested subtree operations (`collectSubtree`,
`duplicateSubtree`, `deleteSubtree`, `reparentKeepWorld`). That makes the
authoring primitives every game needs cheap to build, because a **prefab is a
saved subtree** and the runtime systems compose on top of it.

The north star (per project memory) is a TF2/Overwatch-class networked,
class-based shooter. Rather than build features in the abstract, this stretch is
**game-driven**: a single **wave-survival arena** grows across M70‚ÄìM73 and pulls
each feature into existence. The same two systems the arena needs first ‚Äî
prefabs + spawn points ‚Äî are exactly what the eventual shooter needs for player
and pickup spawning, so this de-risks the long-term vision without committing to
netcode yet.

## Milestones

### M70 ‚Äî Prefab assets (stamp model)
Save a selected subtree to a `.prefab` file and instantiate copies into the
scene. **Stamp semantics**: an instance is a detached copy with no link back to
the file (linked-with-overrides is explicitly deferred ‚Äî see below). Builds
almost entirely on M69's subtree functions + SceneIO. Spec:
`specs/2026-06-14-m70-prefabs-design.md`.

### M71 ‚Äî Spawn points + runtime spawning + world-space nodes
A `SpawnPoint` component (marker: tag/class + which prefab) plus a spawn
system/node that instantiates a prefab at a spawn point's **world** transform
during Play. Folds in the `GetWorldPosition` / `SetWorldPosition` world-space
logic nodes that M69 deferred to "M70/M71" (runtime spawning needs world
placement).

### M72 ‚Äî Navigation core (headless + debug draw)
Nav representation, generation/bake, A\* search, and string-pulling for smooth
paths ‚Äî all pure and unit-testable, plus a debug overlay drawing the nav data
and a computed path. **No gameplay yet.** Decision taken: **grid bake behind a
nav-query interface** (`findPath(start, end) ‚Üí waypoints`). A\*, smoothing,
path-following, and nodes are all "real" and reusable; the query interface lets
a polygon nav mesh be swapped in later without touching agents or nodes. (A
Recast-style auto-baked polygon mesh was considered and deferred as too large
for one milestone.)

### M73 ‚Äî AI agents + nodes + combat
Path-following agent that consumes M72; perception (line-of-sight via the
existing raycaster); node-graph nodes (`MoveTo`, `FindPath`, `HasLineOfSight`,
`Chase`, `Attack`); and the enemy combat loop (chase player, attack, take damage
via the M68 health component, die). Completes the playable wave-survival arena.

## Connecting game ‚Äî wave-survival arena
Lives under `games/` (new project). Enemies spawn from spawn points using an
enemy **prefab**, navigate to the player via the nav mesh, attack, and can be
shot and killed. Exercises prefabs + spawn points + navigation + node-graph
gameplay + UI + physics end-to-end, and is *finishable*.

## Deferred (not in this stretch)
- **Linked prefabs with per-instance overrides** (Unity/UE style) ‚Äî a substantial
  system (override tracking, diff/merge, revert); revisit only if a game demands it.
- **Polygon / Recast-style nav mesh bake** ‚Äî swap-in behind the M72 query interface later.
- **Obstacle avoidance beyond pathing**, networked AI/spawning ‚Äî after the
  single-player arena proves the loop.
